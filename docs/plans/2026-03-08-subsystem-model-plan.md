# SubsystemModel Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Materialize hardware subsystems as queryable objects in the ArchModel, with cross-validated address ranges, replacing the hand-written `RegisterModule` enum with data-driven `SubsystemKind` routing.

**Architecture:** `SubsystemModel` lives on `ModuleModel` (Tile -> Module -> Subsystem hierarchy). Address ranges are derived bottom-up from AM025 register offsets and confirmed top-down from aie-rt extracted constants. A new `gen_subsystems.rs` is generated by `build.rs` and consumed by a new `subsystem_from_offset(offset, TileKind)` function that replaces the old `RegisterModule::from_offset()`.

**Tech Stack:** Rust, xdna-archspec crate, build.rs code generation, existing `gcc -E` aie-rt pipeline.

**Design doc:** `docs/plans/2026-03-08-subsystem-model-design.md`

```rust
/// A functional subsystem within a hardware module.
///
/// Each subsystem owns a contiguous address range within its parent module
/// and contains the registers that control that hardware function.
///
/// Hierarchy: TileTypeModel -> ModuleModel -> SubsystemModel
///
/// Address ranges are cross-confirmed from two independent sources:
/// - AM025 JSON (bottom-up: min/max of member register offsets)
/// - aie-rt module definitions (top-down: explicit base + size)
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub struct SubsystemModel {
    pub kind: SubsystemKind,
    /// Start offset within the tile's 20-bit address space (inclusive).
    pub offset_start: Confirmed<u32>,
    /// End offset (exclusive).
    pub offset_end: Confirmed<u32>,
    /// Indices into parent ModuleModel.registers for registers owned
    /// by this subsystem.
    pub register_indices: Vec<usize>,
}

impl SubsystemModel {
    /// Check whether a tile-local offset falls within this subsystem's range.
    pub fn contains_offset(&self, offset: u32) -> bool {
        offset >= *self.offset_start.value() && offset < *self.offset_end.value()
    }
}
```

**Step 3: Implement `populate_subsystems()`**

Add to `regdb_extractor.rs`:

```rust
/// Group registers by SubsystemKind and derive address ranges.
///
/// Called after all registers are populated on a ModuleModel. Groups
/// registers by their `subsystem` field, computes min(offset) and
/// max(offset + ceil(width/8)) to derive the address range for each
/// subsystem.
pub fn populate_subsystems(module: &mut ModuleModel) {
    use std::collections::BTreeMap;
    use crate::types::{Confirmed, SubsystemModel, SourceAttribution, Source};

    let source = SourceAttribution {
        origin: Source::Am025Json,
        file: "aie_registers_aie2.json".into(),
        detail: format!("register offset range for {:?} module", module.kind),
    };

    // Group register indices by subsystem
    let mut groups: BTreeMap<SubsystemKind, Vec<usize>> = BTreeMap::new();
    for (idx, reg) in module.registers.iter().enumerate() {
        groups.entry(reg.subsystem).or_default().push(idx);
    }

    for (kind, indices) in groups {
        if kind == SubsystemKind::Unknown {
            continue;  // Don't create subsystem for unclassified registers
        }

        let mut min_offset = u32::MAX;
        let mut max_end = 0u32;

        for &idx in &indices {
            let reg = &module.registers[idx];
            let reg_end = reg.offset + (reg.width as u32 + 7) / 8;  // ceil to bytes
            min_offset = min_offset.min(reg.offset);
            max_end = max_end.max(reg_end);
        }

        module.subsystems.push(SubsystemModel {
            kind,
            offset_start: Confirmed::new(min_offset, source.clone()),
            offset_end: Confirmed::new(max_end, source.clone()),
            register_indices: indices,
        });
    }
}
```

**Step 3: Implement cross-validation**

The aie-rt constants are already available at build time via the generated
`gen_aiert_dma.rs` and `gen_aiert_locks.rs`. But these are compile-time
constants in the emulator, not in the archspec crate.

Two approaches:
- (a) Pass aie-rt values into `build_arch_model()` as parameters
- (b) Add aie-rt extraction to the archspec crate itself

Approach (a) is simpler: `build.rs` already has both the ArchModel and the
aie-rt parsed data. After calling `build_arch_model()`, it can call a new
`confirm_subsystem_ranges()` function that walks the model's subsystems
and calls `.confirm()` with the aie-rt values.

Add to `lib.rs` (or a new public function callable from build.rs):

```rust
/// Cross-validate subsystem address ranges against aie-rt extracted constants.
///
/// Called after subsystems are populated from AM025. The aie-rt values come
/// from the gcc -E pipeline in build.rs.
pub fn confirm_subsystem_ranges(
    model: &mut ArchModel,
    confirmations: &[(TileKind, ModuleKind, SubsystemKind, u32, SourceAttribution)],
) {
    for (tile_kind, mod_kind, sub_kind, aiert_base, source) in confirmations {
        let tile = model.tile_types.iter_mut()
            .find(|t| t.kind == *tile_kind)
            .expect("tile type not found");
        let module = tile.modules.iter_mut()
            .find(|m| m.kind == *mod_kind)
            .expect("module not found");
        let subsystem = module.subsystems.iter_mut()
            .find(|s| s.kind == *sub_kind)
            .expect("subsystem not found");

        subsystem.offset_start.confirm(*aiert_base, source.clone());
    }
}
```

Then in `build.rs`, after `build_arch_model()`, call it with the DMA and
lock base addresses from the already-parsed aie-rt data:

```rust
fn gen_subsystems(model: &xdna_archspec::types::ArchModel, out_dir: &Path) {
    use xdna_archspec::types::{TileKind, SubsystemKind};

    let mut out = gen_header("ArchModel subsystem address ranges");

    // Map TileKind to module name for code generation
    let tile_modules: &[(TileKind, &str)] = &[
        (TileKind::Compute, "compute"),
        (TileKind::Mem, "memtile"),
        (TileKind::ShimNoc, "shim"),
    ];

    for &(tile_kind, mod_name) in tile_modules {
        let tile = match model.tile_types.iter().find(|t| t.kind == tile_kind) {
            Some(t) => t,
            None => continue,
        };

        out.push_str(&format!("pub mod {} {{\n", mod_name));

        for module in &tile.modules {
            for sub in &module.subsystems {
                let sub_name = subsystem_mod_name(sub.kind);
                out.push_str(&format!("    pub mod {} {{\n", sub_name));
                out.push_str(&format!("        pub const OFFSET_START: u32 = 0x{:05X};\n",
                    sub.offset_start.value()));
                out.push_str(&format!("        pub const OFFSET_END: u32 = 0x{:05X};\n",
                    sub.offset_end.value()));
                out.push_str("    }\n");
            }
        }

        out.push_str("}\n");
    }

    write_generated(out_dir, "gen_subsystems.rs", &out);
}

fn subsystem_mod_name(kind: SubsystemKind) -> &'static str {
    match kind {
        SubsystemKind::Dma => "dma",
        SubsystemKind::Lock => "lock",
        SubsystemKind::StreamSwitch => "stream_switch",
        SubsystemKind::Processor => "processor",
        SubsystemKind::ProgramMemory => "program_memory",
        SubsystemKind::DataMemory => "data_memory",
        SubsystemKind::Trace => "trace",
        SubsystemKind::Event => "event",
        SubsystemKind::Performance => "performance",
        SubsystemKind::Timer => "timer",
        SubsystemKind::WatchPoint => "watchpoint",
        SubsystemKind::Debug => "debug",
        SubsystemKind::ProgramCounter => "program_counter",
        SubsystemKind::Interrupt => "interrupt",
        SubsystemKind::NoC => "noc",
        SubsystemKind::ShimMux => "shim_mux",
        SubsystemKind::Unknown => "unknown",
    }
}
```

**Step 3: Implement subsystem_from_offset()**

```rust
use crate::archspec::types::{SubsystemKind, TileKind};

/// Determine which hardware subsystem owns a tile-local register offset.
///
/// Uses generated subsystem address ranges from the ArchModel (cross-validated
/// between AM025 and aie-rt). Returns `SubsystemKind::Unknown` for offsets
/// that don't fall within any known subsystem.
pub fn subsystem_from_offset(offset: u32, tile_kind: TileKind) -> SubsystemKind {
    use crate::arch::subsystem;

    // Generate match arms from subsystem constants.
    // Each subsystem range is [OFFSET_START, OFFSET_END).
    match tile_kind {
        TileKind::Compute => {
            // Check each compute subsystem (ordered by offset for clarity)
            if in_range(offset, subsystem::compute::data_memory::OFFSET_START,
                       subsystem::compute::data_memory::OFFSET_END) {
                SubsystemKind::DataMemory
            } else if in_range(offset, subsystem::compute::dma::OFFSET_START,
                              subsystem::compute::dma::OFFSET_END) {
                SubsystemKind::Dma
            } else if in_range(offset, subsystem::compute::lock::OFFSET_START,
                              subsystem::compute::lock::OFFSET_END) {
                SubsystemKind::Lock
            }
            // ... etc for each subsystem
            else {
                SubsystemKind::Unknown
            }
        }
        TileKind::Mem => { /* similar */ }
        TileKind::ShimNoc | TileKind::ShimPl => { /* similar */ }
    }
}

#[inline]
fn in_range(offset: u32, start: u32, end: u32) -> bool {
    offset >= start && offset < end
}
```

Note: The exact set of subsystem modules generated in `gen_subsystems.rs`
depends on what AM025 registers exist per tile type. The match arms must
cover whatever subsystems get generated. If a subsystem has no registers
for a tile type, it won't appear in `gen_subsystems.rs` and shouldn't be
in the match.

**Step 1: Understand the current dispatch pattern**

`state.rs` uses `tile_addr.module()` (which calls `RegisterModule::from_offset()`)
in three functions:

- `read_register()` (line ~322) -- CDO read dispatch
- `read_data()` (line ~429) -- data memory/program memory read

Key mappings:
- `RegisterModule::DmaBufferDescriptor` + `MemTileDmaBufferDescriptor` -> `SubsystemKind::Dma`
- `RegisterModule::DmaChannel` + `MemTileDmaChannel` -> `SubsystemKind::Dma`
- `RegisterModule::Locks` + `MemTileLocks` -> `SubsystemKind::Lock`
- `RegisterModule::StreamSwitch` + `MemTileStreamSwitch` -> `SubsystemKind::StreamSwitch`
- `RegisterModule::CoreModule` -> `SubsystemKind::Processor`
- `RegisterModule::Memory` -> `SubsystemKind::DataMemory`
- `RegisterModule::ProgramMemory` -> `SubsystemKind::ProgramMemory`
- `RegisterModule::MemoryModule` -> needs investigation (misc memory module regs)

**Important:** The current code differentiates `DmaBufferDescriptor` from `DmaChannel`
within the DMA subsystem. After migration, the DMA dispatch will use SubsystemKind::Dma
and then internally distinguish BD vs channel vs status by offset sub-ranges. This
sub-dispatch can use the existing `regdb::device_reg_layout()` values.

## Notes

- The MemoryModule misc region (0x1E000-0x1EFFF) needs attention during migration.
  Check which SubsystemKind the AM025 registers in this range get tagged with
  by `regdb_extractor`. It may be Event, Trace, or a mix -- the subsystem
  classification will tell us.

- `RegisterInfo.module` field currently stores `RegisterModule`. During Task 7,
  decide whether to change it to `SubsystemKind` or remove it. Since
  `RegisterInfo` is used for display/debugging (not routing), changing the
  field type is sufficient.

- Stream switch registers appear at different offset ranges per tile type
  (compute: 0x3F000, memtile: 0xB0000). The SubsystemModel handles this
  naturally since each tile type gets its own subsystem with its own range.

- The `from_offset()` function currently has no tile type context, which is
  why it has separate compute and memtile variants. The new
  `subsystem_from_offset(offset, TileKind)` eliminates this problem.
