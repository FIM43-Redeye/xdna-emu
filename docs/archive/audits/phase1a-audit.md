## Type Boundary

### `TileType` (runtime side)

Defined in `src/device/tile/core_state.rs` at line 49:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum TileType {
    Shim,
    MemTile,
    Compute,
}
```

Three variants.  Has inherent methods `is_shim()`, `is_mem_tile()`,
`is_compute()`.  No `Display`, no `Serialize`/`Deserialize`, no `Hash`.

Re-exported chain:
`core_state::TileType`
→ `tile::mod.rs` (`pub use core_state::{..., TileType, ...}`)
→ `device::mod.rs` (`pub use tile::{..., TileType, ...}`)

The `ArchConfig` trait takes `TileType` as a method parameter and return type
for 10 of its 20 methods.

There is a second, unrelated `TileType` in `src/trace/vcd.rs` (line 54) that
has the same three variant names but is entirely local to the VCD parser and
not involved in `ArchConfig`.

### `TileKind` (archspec crate)

Defined in `crates/xdna-archspec/src/types.rs` at line 62:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub enum TileKind {
    Compute,
    Mem,
    ShimNoc,
    ShimPl,
}
```

Four variants.  Has `Display`.  Has `Serialize`/`Deserialize` and `Hash`.

### Structural Comparison

| Property | `TileType` | `TileKind` |
|----------|-----------|-----------|
| Variants | 3: `Shim`, `MemTile`, `Compute` | 4: `ShimNoc`, `ShimPl`, `Mem`, `Compute` |
| Shim model | Single `Shim` variant | Split `ShimNoc` / `ShimPl` |
| Mem tile name | `MemTile` | `Mem` |
| Compute name | `Compute` | `Compute` |
| `Hash` | No | Yes |
| `Serialize`/`Deserialize` | No | Yes |
| Inherent methods | `is_shim()`, `is_mem_tile()`, `is_compute()` | None (has `Display`) |

The two types are **not structurally identical**.  Key differences:

1. **Extra variant**: `TileKind` has `ShimPl` (ShimPL tiles exist in the
   hardware but are not software-accessible on NPU devices).  `TileType` has
   no `ShimPl` equivalent — it treats all row-0 tiles as `Shim`.

2. **Name mismatch**: `TileType::MemTile` vs `TileKind::Mem`.

3. **Derive set**: `TileKind` derives `Hash`, `Serialize`, `Deserialize`;
   `TileType` does not.

### Recommendation

`TileType` cannot be a plain type alias for `TileKind` because of the extra
`ShimPl` variant.  The runtime only ever sees `ShimNoc` tiles on NPU1 (all
row-0 tiles are NoC-connected); `ShimPl` does not appear in the emulator's
tile array.

**Recommended path (for Task 2):** Replace `TileType` with `TileKind` at the
`ArchConfig` trait boundary.  The `ArchConfig::tile_type()` implementation in
`ModelConfig` can return `TileKind::ShimNoc` for row 0 (matching the device
model extractor, which already distinguishes them).  Callers that pattern-match
on `TileType::Shim` will need to match both `TileKind::ShimNoc` and
`TileKind::ShimPl`, or the emulator can declare that `ShimPl` is never used
and add an `unreachable!()` branch.  The latter is simpler and matches current
behavior.

If complete replacement is too disruptive for Task 2, a narrower option is to
keep `TileType` as a thin runtime projection of `TileKind` (a 3-way enum
derived from the 4-way one), with a `From<TileKind>` conversion that maps both
`ShimNoc` and `ShimPl` to `TileType::Shim`.  This avoids churning all
`ArchConfig` consumers in Task 2 and defers the full rename to a later pass.

## Port Data Dependency

### What `crate::arch` is

`crate::arch` is a module declared in `src/lib.rs` (line 61) that consists of
`include!()` macros pointing at files generated into `$OUT_DIR` by `build.rs`.
It is **not a source file** — it has no persistent `.rs` file on disk.  The
generated files are:

| File | Included at | Content |
|------|-------------|---------|
| `gen_arch.rs` | `mod arch { include!(...) }` | Topology constants and per-tile resource counts for NPU1 (`COLUMNS`, `ROWS`, `NUM_MEM_TILE_ROWS`, `MAX_LOCK_VALUE`, `compute::*`, `memtile::*`, `shim::*`) |
| `gen_stream_ports.rs` | `mod arch { include!(...) }` | `COMPUTE_MASTER_PORTS`, `COMPUTE_SLAVE_PORTS`, `MEMTILE_MASTER_PORTS`, `MEMTILE_SLAVE_PORTS`, `SHIM_MASTER_PORTS`, `SHIM_SLAVE_PORTS` as `&'static [u8]` |
| `gen_stream_ranges.rs` | `mod arch { pub mod stream_switch { include!(...) } }` | Sub-modules `compute`, `mem_tile`, `shim` each containing `NORTH_MASTER_START/END`, `SOUTH_MASTER_START/END`, `NORTH_SLAVE_START/END`, `SOUTH_SLAVE_START/END`, `EAST_*`, `WEST_*` as `u8` constants |
| `gen_subsystems.rs` | `mod arch { pub mod subsystem { include!(...) } }` | Per-tile-type subsystem address ranges |

### Generation chain for port data

1. `build.rs` calls `gen_stream_ports(regdb, out_dir)` (line 149 of build.rs).
2. `gen_stream_ports()` (build.rs line 946) calls `collect_port_array()` which
   reads `Stream_Switch_Master_Config_*` and `Stream_Switch_Slave_Config_*`
   register names from the AM025 register database (`RegisterDb`, loaded from
   `mlir-aie/lib/Dialect/AIE/Util/aie_registers_aie2.json`).
3. The per-port-index type is inferred from the register name suffix (e.g.,
   `Core0` → `PT_CORE`, `FIFO0` → `PT_FIFO`, `North0` → `PT_NORTH_BASE + 0`).
   This name-to-type mapping is a convention defined in `build.rs`, not in the
   hardware spec.
4. `gen_stream_ranges()` (line 1127) post-processes the arrays produced by
   `gen_stream_ports()` to find the first and last port index of each
   directional group, generating the `*_START`/`*_END` constants.

The source of the port data is the **AM025 register database JSON**
(`aie_registers_aie2.json` from mlir-aie), the same source that drives BD
field parsing.  The data is **AIE2-specific**: the register names encode the
port topology for AIE2 tile types.

### What moving `ArchConfig` into the crate requires

The four port-layout methods (`master_ports`, `slave_ports`,
`north_master_range`, `south_master_range`, `north_slave_range`,
`south_slave_range`) return `&'static [u8]` pointing at data in `crate::arch`,
which is a generated, NPU1-specific module.  These static arrays are
intentionally NPU1-only — the code comment (arch_config.rs line 419) says:
"When AIE2P diverges in port layout, we'll add AIE2P-specific arrays and
select based on architecture."

Two options for Task 3 (moving `ArchConfig` into the crate):

**Option A — Keep port-layout methods as a thin runtime-side extension trait.**
Define a separate `PortLayout` trait (or a blanket provided-method group) on
the runtime side that augments `ArchConfig` with port-data access.  The core
`ArchConfig` trait moves into `xdna_archspec` without the six port-layout
methods; the runtime crate adds them back via a sub-trait or inherent impl on
`ModelConfig`.  This avoids pulling the AM025 JSON generation into the crate.

**Option B — Move port-data generation into the crate.**
Add a `build.rs` to `xdna-archspec` that generates the same arrays.  The
workspace crate then exports them as part of its public API.  This is the
cleaner long-term design but is a larger build-system change.

**Recommendation for Task 3:** Use Option A.  The port-layout methods are the
only `ArchConfig` methods that depend on `crate::arch`; all other methods
use only `self` fields populated from `ArchModel`.  Keeping port-layout as a
runtime-side extension isolates the build.rs coupling and lets `ArchConfig`
(the core trait) move cleanly into `xdna_archspec` without a build.rs rewrite.
The extension trait can be collapsed later once AIE2P port data is needed and
the generation is worth moving.

One additional complication: `program_memory_size()` for the `Compute` variant
also directly references `crate::arch::compute::PROGRAM_MEMORY_SIZE` (line
392).  However, this value is also present in `ArchModel` via
`TileTypeModel::memory::program_memory_bytes` (defined in
`crates/xdna-archspec/src/types.rs`), so the runtime-side implementation
should add a dedicated `program_memory_size` field to `TileTypeParams`,
populated from `memory.program_memory_bytes`, and use that field instead of
the generated constant.  Note: do NOT reuse `data_memory_size` for program
memory -- they are different values (64KB data vs 16KB program on AIE2
compute tiles).  With this field added, the only non-port dependency on
`crate::arch` is removed from the trait implementation, making Option A
fully viable.

## TileType vs TileKind Resolution

Task 2 outcome.

The two enums are structurally incompatible (3 vs 4 variants; name mismatch
`MemTile`/`Mem`; extra `ShimPl` that has no runtime equivalent), so a plain
type-alias cannot be used.

**Resolution:** `From<TileKind> for TileType` and `From<TileType> for TileKind`
added in `src/device/tile/core_state.rs` (commit `43fc807`). `ShimPl` collapses
to `Shim` (lossy, documented, and tested). `TileType::Shim` promotes to
`TileKind::ShimNoc` (emulator invariant: no `ShimPl` tiles exist in the array).

Deep rename (`TileType::MemTile` -> `Mem`, etc.) is deferred to the Tile
Topology subsystem pass (Subsystem 2). `ArchConfig` will take/return `TileKind`
at its interface after Task 3 moves the trait into the crate; runtime callers
using `TileType` convert via `.into()` at the boundary.
