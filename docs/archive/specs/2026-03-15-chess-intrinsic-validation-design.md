# Chess Intrinsic Validation Path

## Purpose

Add a Chess-native intrinsic path to the instruction-level validation harness.
The existing Peano path covers 76 intrinsics derived from IntrinsicsAIE2.td
(LLVM builtins). The Chess path parses `me_chess_opns.h` (Synopsys ISG-generated
header with ~1,700 function declarations) to cover the full breadth of Chess
intrinsics -- including both low-level hardware primitives and user-facing
wrapper functions.

The real NPU remains the oracle. We don't compute expected values -- we verify
that HW and EMU agree.

## Motivation

The Peano path has structural limitations:

1. **76 of 316 intrinsics** -- exclusions for i64 ABI ambiguity, constant-index
   operations, IntrInaccessibleMemOnly (config register preamble), and
   multi-return types leave large gaps in coverage.
2. **ClangBuiltin names are Peano-only** -- `__builtin_aiev2_*` names from
   IntrinsicsAIE2.td do not work with Chess. Chess uses completely different
   intrinsic names (e.g., `broadcast_to_v16int32` instead of
   `__builtin_aiev2_vbroadcast32_I512`).
3. **Chess is ground truth** -- it is the production compiler with complete
   intrinsic coverage (~1,700 declarations vs 76 Peano builtins).

The Chess path is independent of the Peano path. Both exist side by side.
This is a dual-path design, not a merger.

## Architecture

```
me_chess_opns.h (55K lines, ~1,700 declarations)
       |
  Pre-processor: extract chess_property/chess_storage annotations
       |
  Cleaned C++ + annotations side table
       |
  Type stubs from me_chess_types.h (chessTraitsOf<T> -> sized structs)
       |
  clang.cindex AST parse (full semantic analysis)
       |
  AST walker: extract ChessIntrinsic records
       |
  Filter (skip volatile, dont_care, void, unsized, operators, etc.)
       |
  tools/chess-test-gen.py (generate kernel.cc + aie.mlir per intrinsic)
       |
  scripts/instr-test.sh --chess (compile with xchesscc, run, compare)
       |
  Per-intrinsic: PASS (HW == EMU) or FAIL (divergence)
```

## Source Files

### Primary: `aietools/data/aie_ml/lib/isg/me_chess_opns.h`

55,087 lines, auto-generated by Synopsys noodle compiler. Contains:

- **~1,627 `me_primitive` declarations** -- raw hardware primitives, each in
  its own `namespace me_primitive { ... }` block. Use low-level types like
  `v512w8`, `w128`, `pmode_t`.
- **~93 inline wrapper functions** -- user-facing API like `mul_2x8_8x8`,
  `broadcast_to_v16int32`, `insert`. Use standard vector types (`v16int32`,
  `v64int8`). Often call me_primitive functions with computed config words.
- **Operator overloads** -- `operator+=`, `operator-=`, etc. on vector types.
- **Diagnostic/utility functions** -- `chess_report`, `chess_assert`, etc.

Both layers (me_primitive and inline wrappers) are in scope.

### Type Reference: `aietools/data/aie_ml/lib/isg/me_chess_types.h`

18,170 lines. Defines ~167 types as C++ classes with `VBit`/`VBitVector`
templates. Key categories:

| Category | Count | Examples |
|----------|-------|---------|
| Vector int/uint | 64 | v16int32, v64uint8, v128int8 |
| Accumulator vectors | 12 | v8acc32, v16acc64, v32accfloat |
| Complex vectors | 19 | v4cint16, v8cint32, v8cfloat |
| Float/bfloat vectors | 15 | v16float, v32bfloat16 |
| Sparse/compressed | 22 | v256uint4_sparse, v16int32_compress |
| Scalar/special | 35 | mask64, mod_t, addr_t, pmode_t, bfloat16 |

Size metadata is available via `chessTraitsOf<T>` specializations (lines
~17900-18120), each providing `bits` and `elems`.

## Components

### 1. Pre-processor: Chess Extension Extraction

Pure function: `(header_text) -> (clean_cpp, annotations)`.

Strips Chess-specific extensions from `me_chess_opns.h` while recording
semantic annotations in a side table keyed by `(function_name, line_number)`.

**Stripped and recorded (preserved as metadata):**

| Pattern | Action | Annotation |
|---|---|---|
| `chess_property(X)` | Strip, record X | `properties: [X, ...]` |
| `chess_storage(X)` on parameters | Strip, record X | `storage_params: [X, ...]` |

**Stripped and discarded (body-level or irrelevant):**

| Pattern | Action |
|---|---|
| `chess_protect_access` | Strip |
| `chess_manifest(...)` | Strip (body-level) |
| `chess_memory_fence()` | Strip (body-level) |
| `chess_separator_scheduler()` | Strip (body-level) |
| `chess_dont_warn_dead(...)` | Strip |
| `chess_unroll_loop(*)` | Strip |
| `VBITzCONSTEXPR` | Replace with `inline` |
| `#if 0//!` ... `#endif//!` | Remove entire block |
| `//!` comment lines | Remove |

The annotation data is preserved so future work can act on Chess semantics
(e.g., `chess_property(functional)` as a positive testability signal,
`chess_property(commutative)` for operand-swap test generation).

**Property matching**: Properties are stored as the full text inside the
parentheses (e.g., `"guarded_memory_fence volatile output_stage_offset_7"`).
Filters use word-level matching: the property string is split on whitespace,
and each filter keyword is checked against the resulting word list. This means
`"volatile" in words` correctly matches both `chess_property(volatile)` and
`chess_property(guarded_memory_fence volatile ...)`.

**Chess Property Catalogue (22 unique variants observed):**

| Property | Count | Meaning | Filter Action |
|---|---|---|---|
| `volatile` | 93 | Side effects (acquire, release, event) | Skip |
| `dont_care` | 77 | Return undefined (undef_*) | Skip |
| `non_functional` | 48 | Operator overloads with side effects | Skip |
| `keep_with_operand` | 31 | Cascade/SCD operations | Skip |
| `arg_mem_only inlined_at_sfg` | 6 | Lock internals | Skip |
| `guarded_memory_fence volatile ...` | 4 | Fence + volatile | Skip |
| `do_generate` | 36 | Type conversions | Testable |
| `functional loop_free` | 26 | Pure, no loops | Testable |
| `functional` | 4 | Pure | Testable |
| `commutative` | 7 | Commutative math | Testable |
| `select` | 9 | Conditional ops | Testable |
| `do_generate_llvm` | 8 | LLVM codegen hint | Testable |
| `keep_in_registers` | 8 | Struct placement hint | Ignore |
| `indexed` | 2 | Access pattern hint | Ignore |
| `pointer_template` | 1 | Template hint | Ignore |
| `duplicate_at_using_opn2` | 1 | Encoding hint | Ignore |

**Estimated filter impact**: The skip categories total ~259 declarations
(93 + 77 + 48 + 31 + 6 + 4). Additional skips for void return, unsized
types, operators, and diagnostics will further reduce the testable set.
Rough estimate: **~800-1,200 testable intrinsics** after filtering. The
exact count will be determined during implementation by running the
generator in `--dry-run` mode.

### 2. Type Stubs for clang.cindex

clang.cindex needs type resolution. Rather than parsing the full 18K-line
`me_chess_types.h` (which depends on Chess compiler internals), we generate
a minimal stub header with correctly sized structs.

**Source**: `chessTraitsOf<T>` specializations at the end of me_chess_types.h.
Each provides `static const unsigned bits = N;` -- divide by 8 for byte size.

**Output**: `chess_type_stubs.h` (auto-generated):
```cpp
// Auto-generated from me_chess_types.h chessTraitsOf<T> specializations
struct v16int32 { char _data[64]; };   // 512 bits, 16 elems
struct v8acc64  { char _data[64]; };   // 512 bits, 8 elems
struct bfloat16 { char _data[2]; };    // 16 bits
struct mask64   { char _data[8]; };    // 64 bits
// ... ~167 types
```

**Generation**: A small extraction script (`chess_type_stubs.py`) parses the
chessTraitsOf block and emits the stub header. This runs once (or whenever
aietools updates).

**Future**: If we need real type semantics (conversion operators, accumulator
behavior, element access), we replace stubs with proper integration of the
full me_chess_types.h type hierarchy. The stub generator is isolated for easy
swap-out.

### 3. AST Walker: Function Extraction

Feeds cleaned header + type stubs to clang.cindex. Walks the AST to extract
a list of `ChessIntrinsic` records:

```python
@dataclass
class ChessIntrinsic:
    name: str                          # "broadcast_to_v16int32"
    namespace: str                     # "me_primitive" or "" (global)
    return_type: str                   # "v16int32"
    return_size: int                   # bytes, from AST type info
    params: list[tuple[str, int]]      # [(type_name, size_bytes), ...]
    is_inline: bool                    # has body (wrapper) vs declaration
    properties: list[str]              # from pre-processor annotations
    storage_params: list[str]          # chess_storage on any parameter
    overload_index: int                # disambiguator for same-name functions
    source_line: int                   # traceability back to me_chess_opns.h
```

**Overload handling**: Functions with identical names but different signatures
are common in Chess (e.g., `mul_2x8_8x8` has 4+ variants). Each overload
becomes a separate test with a directory name encoding the signature:
- `mul_2x8_8x8__v32int16_v64uint8/` (2-arg variant)
- `mul_2x8_8x8__v32int16_int_v64uint8_int/` (4-arg signed variant)
- `mul_2x8_8x8_conf__v32int16_v64uint8_int_int/` (conf variant)

The directory name is `{func_name}__{param_types_joined_by_underscore}`.
This encoding is stable across regenerations of me_chess_opns.h (unlike
index-based suffixes which depend on source ordering). The manifest also
records the full parameter signature for unambiguous identification.

**Error handling**: If clang.cindex fails to parse a section of the cleaned
header (e.g., due to an unhandled Chess extension that survived
pre-processing), the walker logs a warning with the source location and
skips the unparseable declaration. The manifest records it as skipped with
reason `"parse_error: <diagnostic>"`. The generator does not fail hard on
individual parse errors -- it processes what it can and reports failures.

### 4. Filter

Applied after AST extraction. Each function is classified as testable or
skipped with a documented reason.

**Skip criteria:**

| Filter | Rule | Source |
|---|---|---|
| Undefined return | `"dont_care" in property_words` | chess_property |
| Side effects | `"volatile" in property_words` | chess_property |
| Non-functional | `"non_functional" in property_words` | chess_property |
| Cascade/SCD | `"keep_with_operand" in property_words` | chess_property |
| Lock internals | `"arg_mem_only" in property_words` | chess_property |
| Memory-bank params | `len(storage_params) > 0` | chess_storage |
| Void return | `return_type == "void"` | AST |
| Unsized types | `return_size == 0 or any param size == 0` | AST |
| Operator overloads | `name.startswith("operator")` | AST |
| Diagnostic functions | `name.startswith("chess_report") etc.` | Name pattern |

Note: `property_words` is the chess_property string split on whitespace.
This ensures `"volatile"` matches both `chess_property(volatile)` and
multi-word properties like `chess_property(guarded_memory_fence volatile ...)`.

The `chess_storage` filter affects only functions with storage-annotated
**parameters** (e.g., `char chess_storage(TM) *mem`). Most chess_storage
usage (~1,000 of 1,051 occurrences) is on global variables and function
bodies, not parameter types, so this filter is narrow. The exact count
of affected function declarations will be reported in the manifest.

**Positive signals (recorded, not used for filtering yet):**
- `"functional"` or `"functional loop_free"` in property_words
- `"commutative"` in property_words
- `"do_generate"` in property_words

### 5. Kernel Code Generation

Generated kernel.cc for Chess path:

```cpp
// Auto-generated: tests broadcast_to_v16int32 (Chess)
// Signature: v16int32 = f(int)
// Namespace: me_primitive
#define NOCPP
#include <stdint.h>

extern "C" {
void test_kernel(const int32_t *restrict in, int32_t *restrict out) {
    int arg0 = in[0];
    v16int32 result = broadcast_to_v16int32(arg0);
    v16int32 *out_vec = (v16int32 *)out;
    *out_vec = result;
}
} // extern "C"
```

**Differences from Peano path:**
- Chess-native function names (no `__builtin_aiev2_` prefix)
- No include for intrinsic declarations -- xchesscc knows them natively
- `#define NOCPP` suppresses C++ standard library inclusion (same as Peano);
  xchesscc sets `__AIENGINE__` and `__AIEARCH__` automatically based on
  target architecture, so we do not define them in the kernel
- Chess type names used directly (same for common types, additional types
  like mask64, complex vectors, sparse types available)
- For me_primitive functions, called unqualified -- xchesscc resolves them
  natively. If bare names fail to compile during implementation, the
  fallback is to emit `using namespace me_primitive;` at the top of the
  kernel. This will be validated in the first implementation task.

**Shared with Peano path (identical):**
- MLIR template (link_with = "kernel.o", objectfifo, runtime_sequence)
- Host harness (test_host.cpp, PRNG, XRT interface)
- Manifest JSON format (with added `compiler`, `namespace`, `properties` fields)
- Input data generation (same LCG PRNG, same seed)

**test_host.cpp management**: Each generator produces its own copy of
test_host.cpp in its output directory. The content is identical -- the
Peano generator's `generate_test_host_cpp()` function is imported and
reused by the Chess generator.

### 6. Runner Integration

`scripts/instr-test.sh` gains compiler selection flags:

| Flag | Generator | Kernel Compile | MLIR Compile |
|---|---|---|---|
| `--peano` (default) | instr-test-gen.py | Peano clang --target=aie2 | aiecc --no-xchesscc --no-xbridge |
| `--chess` | chess-test-gen.py | xchesscc_wrapper aie2 | aiecc --xchesscc --xbridge |
| `--both` | Both generators | Both compilers | Both linkers |

**Compilation detail**: Both paths use a two-step compile:
1. Kernel compile: `kernel.cc -> kernel.o` (Peano clang or xchesscc)
2. MLIR compile: `aie.mlir -> aie.xclbin + insts.bin` (aiecc.py with
   appropriate flags). The `link_with = "kernel.o"` attribute in the MLIR
   tells aiecc.py to link the pre-compiled kernel object. This mechanism
   works with both `--xchesscc --xbridge` (Chess linker) and
   `--no-xchesscc --no-xbridge` (Peano linker) -- the kernel.o just needs
   to be in the same directory as aie.mlir.

**Output directories:**
- `build/instr-tests-peano/` (relocated from current `build/instr-tests/`)
- `build/instr-tests-chess/`

The rename from `build/instr-tests/` to `build/instr-tests-peano/` is a
one-time migration. The old directory can be deleted (it is gitignored build
output, trivially regenerated). The `--peano` flag sets
`OUT_DIR=build/instr-tests-peano/`.

**Result directories** (per-compiler, not shared):
- `/tmp/instr-test-results-YYYYMMDD-chess/`
- `/tmp/instr-test-results-YYYYMMDD-peano/`

Result files within each directory use the existing naming convention
(`<name>_hw.bin`, `<name>_emu.bin`) without compiler qualifiers, since the
directories are already separated. This avoids any changes to how the Peano
path names its output files.

Phases (generate, compile, HW run, EMU run, compare, report) are unchanged.
Only the compile commands and directory paths differ per compiler path.

## Scope

**In scope (initial):**
- Both me_primitive and inline wrapper functions from me_chess_opns.h
- clang.cindex AST parsing with pre-processed header
- Type stubs generated from chessTraitsOf<T>
- Chess extension annotation extraction and filter-based classification
- xchesscc compilation path
- Binary exact HW vs EMU comparison
- Single seed per test

**Out of scope (future):**
- Full me_chess_types.h integration (replace stubs with real type hierarchy)
- Acting on chess_property semantics beyond filtering (e.g., commutative
  operand-swap tests, functional purity guarantees)
- IntrInaccessibleMemOnly intrinsics (config register preamble)
- Sparse/compressed type test inputs (need special encoding)
- Multi-seed / fuzz mode
- Floating-point tolerance comparison
- Cross-compiler comparison (same intrinsic on Chess vs Peano)

## Success Criteria

1. Pre-processor extracts annotations without data loss and produces
   parseable C++ for clang.cindex
2. Type stubs cover all types referenced in me_chess_opns.h function
   signatures
3. AST walker extracts all function declarations including overloads
4. Filter correctly classifies testable vs skipped intrinsics with
   documented reasons
5. Generated kernel.cc files compile cleanly with xchesscc
6. End-to-end: at least one Chess intrinsic runs on both HW and EMU,
   producing comparable output
7. Manifest tracks coverage: every declaration is either generated or
   has a documented skip reason
8. Existing Peano path behavior is preserved; only change is the output
   directory moving from `build/instr-tests/` to `build/instr-tests-peano/`
   (a one-time migration of gitignored build output)
