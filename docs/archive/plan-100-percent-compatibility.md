# Path to 100% Emulator Compatibility

Status: ACTIVE PLAN (2026-02-19)

This document captures everything we know about achieving full hardware-accurate
emulation of AMD XDNA NPUs, organized by what we need, what sources exist, and
what work remains.

## The Four Pillars

Full emulator accuracy requires four independent capabilities. Each has
different data sources and different levels of remaining work.

---

## Pillar 1: Instruction Decoding

**Question**: Given 128 bits of VLIW, what instructions are present and what
are their operands?

### Sources (in order of authority)

1. **ISB binary** (`aietools/data/aie_ml/lib/isg/me.isb`, 4.3MB)
   - THE authoritative source. Generated by Synopsys `noodle` from nML.
   - Contains 101 enums, 44 format templates, 293 bit patterns, 251 encodings.
   - Binary TLV format, partially reverse-engineered.
   - Parser: `tools/isb_parser.py`
   - Status: ~60% parsed (enums, formats, patterns extracted; operand binding
     and encoding block internals still being decoded).

2. **llvm-aie TableGen** (`llvm-aie/llvm/lib/Target/AIE/`)
   - 683 encoding definitions across 7 VLIW slots.
   - Open-source, machine-readable, already driving our decoder.
   - **Possibly incomplete**: our decoder fails on some Chess-compiled binaries.
   - The Peano compiler has known bugs, suggesting the TableGen may lag the
     real hardware.

3. **Chess assembler as oracle**
   - `xchesscc --aiearch aie2 -p me +a <asm>` assembles any instruction.
   - Can generate ground-truth encoding for any specific instruction.
   - Useful for validation, not for bulk enumeration.

4. **Chess .lst files**
   - Every Chess build generates .lst with hex bytes + mnemonic.
   - Ground truth for specific binaries, not a general encoding database.

### Remaining Work

- [ ] Complete ISB parser: resolve global ID namespace, operand bindings,
      encoding block internal format
- [ ] Cross-validate ISB against llvm-aie TableGen (find what llvm-aie misses)
- [ ] Fix specific decoder failures (use .lst as ground truth for each)
- [ ] Build a comprehensive encoding test: assemble every instruction via Chess,
      feed bytes to decoder, verify match

### Architecture Notes

- ISB uses a global ID namespace for all entities (enums, operand groups,
  encoding blocks, scheduling resources)
- Known ID ranges: 15870-16281 (enums), 18982-19202 (bundle scheduling),
  19203-19431 (decode/resource scheduling)
- Unresolved operand IDs: 16351 (most common, 55 occurrences), 17037, 17036,
  8742, 16543
- Operand group names (mAluCg, mLdaCg, mMvSclDstCg) are llvm-aie conventions,
  NOT Synopsys names -- the ISB uses numeric IDs internally

## Pillar 2: Instruction Semantics

**Question**: Given a decoded instruction, what output values does it produce?

### Sources (in order of authority)

1. **Real NPU hardware** (ground truth)
   - Run instruction on hardware, observe output.
   - Our test runner already does this for end-to-end tests.
   - Cannot isolate individual instruction semantics easily.

2. **Python vector models** (`aietools/data/aie_ml/lib/python_model/model/`)
   - Step-by-step reference implementations of hardware vector operations.
   - `mulmac.py` (17KB): VMAC, VMULT -- multiply-accumulate with all modes.
   - `srs.py` (3.6KB): Shift-right-and-saturate.
   - `ups.py` (577B): Unpack-and-shift.
   - `constants.py` (30KB): Rounding modes, UPS/SRS tables, VADD mode tables,
     saturation constants. This file alone is a treasure trove.
   - `bfloat16.py` (14KB): bfloat16 arithmetic.
   - `pack.py` (1.1KB): Data packing/unpacking.
   - `ldst.py` (1.8KB): Load/store address generation (2D/3D).
   - `helpers.py` (1.9KB): Truncation, sign extension utilities.
   - These exist ONLY for AIE2, not AIE2P or AIE1.

3. **Configuration word constants** (`aietools/data/aie_ml/lib/isg/me_chess.ctt`, 13MB)
   - Every `#const` definition for shuffle modes, multiply modes, sign modes,
     rounding modes, etc.
   - These are the numeric values that appear in vector instruction config words.
   - Must-have for implementing correct mode selection.

4. **Intrinsic signatures** (`aietools/data/aie_ml/lib/isg/me_chess_opns.h`, 2.1MB)
   - Every C++ intrinsic function signature for all instructions.
   - Tells us what operations exist and their argument/return types.
   - Does not tell us the computational semantics.

5. **Type conversion table** (`aietools/data/aie_ml/lib/isg/me_chess.tof`, 749KB)
   - 11,700 type signatures showing all legal conversions between vector,
     scalar, accumulator, and pointer types.
   - Useful for implementing type coercion rules.

6. **AM020/AM025 documentation** (`docs/xdna/`)
   - Architecture reference manuals (extracted text, not PDFs).
   - Covers areas not in Python models (scalar ops, control flow, etc.).

### Remaining Work

- [ ] Consume Python model files and implement vector compute semantics
  - Priority: VMAC (most complex, most used), SRS, UPS, VADD/VSUB
  - Each model file is a self-contained reference implementation
  - Read, understand the hardware behavior, write original Rust code
- [ ] Parse me_chess.ctt and build configuration word lookup tables
- [ ] Implement scalar ALU operations from AM020 descriptions
- [ ] Implement remaining control flow (interrupts, events, etc.)

### Current Coverage

- Scalar ALU: basic ops working (add, sub, and, or, xor, shift, compare)
- Control flow: branches, calls, loops, delay slots all working
- Vector compute: ~10% implemented (basic VADD, minimal VMAC)
- DMA address generation: working (matches hardware)
- Lock operations: working

## Pillar 3: Pipeline Timing

**Question**: When does each operation start, how long does it take, and what
can execute in parallel?

### Sources

1. **ISG scheduling files** (newly discovered in aietools)
   - `me_chess_bndl.isg` (107KB): 313 bundle formation rules, pipeline stages
     (fst/ist/ost = first/input/output slot), resource constraints, instruction
     IDs. RIDs 18982-19202.
   - `me_chess_dr.isg` (67KB): 266 decode/resource records, data dependencies,
     forwarding paths, hazard windows. RIDs 19203-19431.
   - `me_chess.sfg` (27KB): Stage/format generation rules.
   - `me_chess_bndl.sfg` (15KB): Bundle code selection patterns.
   - `me_chess_dr.sfg` (33KB): Delay/reservation scheduling rules.

2. **llvm-aie scheduling model**
   - 278+ itinerary classes with cycle counts.
   - Already parsed by our TableGen infrastructure.
   - Cross-validates against the ISG scheduling files.

3. **Register layout** (`me.rlo`, 35KB)
   - Authoritative register file structure from Synopsys `animal` tool.
   - Register widths, port counts, memory bank sizes, address classes.

4. **Connectivity report** (`me_connectivity.rpt`, 27KB)
   - Type-to-register mapping.
   - Shows which types can live in which registers.
   - Promotion/conversion rules between register classes.

### Remaining Work

- [ ] Build pipeline model from ISG fst/ist/ost values
- [ ] Implement resource conflict detection (rsc fields)
- [ ] Model data forwarding based on DR dependency graph
- [ ] Validate against llvm-aie itinerary cycle counts
- [ ] Implement VLIW slot constraints accurately

### Priority: LATER

Cycle-accuracy is Phase 2 work. Functional correctness comes first. The data
is here when we need it, but it does not block any current work.

## Pillar 4: Hardware Subsystems

**Question**: How do DMA, locks, stream switches, memory, and shim tiles
behave?

### Sources

1. **aie-rt** (`mlir-aie/third_party/aie-rt/driver/src/`)
   - The same library that programs real silicon. Authoritative for:
     - DMA BD programming, channel start, completion polling
     - Lock acquire/release/set/get
     - Stream switch circuit/packet configuration
   - Already our primary reference, well-consumed.

2. **AM025 register database** (`mlir-aie/lib/Dialect/AIE/Util/aie_registers_aie2.json`)
   - 1,806 registers, 6,412 fields. Fully consumed by regdb.rs.
   - BD field parsing is 100% data-driven from this source.

3. **Device model** (`tools/aie-device-models.json`)
   - Array dimensions, tile types, memory sizes, lock counts, BD counts,
     DMA channel counts, stream switch port counts.
   - Fully consumed at startup.

4. **Module ISS interface** (`me_module_iss.isg`, 12KB)
   - 140+ transitory signals with types.
   - Memory access patterns, DMA interfaces, lock signals, debug interfaces.
   - Useful for understanding module-level I/O behavior.

5. **Checkers config** (`me_module_checkers_config.txt`, 22KB)
   - Simulator I/O specs, 80+ built-in simulation functions.
   - Lock/memory interface specifications.

### Remaining Work

- [ ] Control packet handling (partially working, some edge cases)
- [ ] DMA task chaining with tokens
- [ ] Memtile DMA (larger address space, more channels)
- [ ] Stream switch packet routing (circuit routing works)
- [ ] Event/interrupt system
- [ ] Timer synchronization

### Current Coverage

DMA, locks, circuit-switched streams, and basic memory operations are all
working. Hardware comparison tests validate end-to-end correctness for 10+
test cases. The subsystem layer is the most mature part of the emulator.

## Toolchain Reference Summary

All of these live in aietools, NOT in the AMD unified software bundle (which
is 175GB of Vivado/FPGA tooling with essentially nothing useful for us).

### Files that matter (aietools/data/aie_ml/lib/)

| File | Size | Pillar | Priority | Content |
|------|------|--------|----------|---------|
| `isg/me.isb` | 4.3MB | Decoding | HIGH | Complete instruction encoding database |
| `python_model/model/*.py` | ~75KB | Semantics | HIGH | Vector compute reference implementations |
| `isg/me_chess.ctt` | 13MB | Semantics | HIGH | Configuration word constants |
| `isg/me_chess_opns.h` | 2.1MB | Semantics | MEDIUM | Intrinsic signatures |
| `isg/me_chess_bndl.isg` | 107KB | Timing | LATER | Bundle scheduling rules |
| `isg/me_chess_dr.isg` | 67KB | Timing | LATER | Resource dependencies |
| `isg/me.rlo` | 35KB | Timing | LATER | Register layout |
| `isg/me_connectivity.rpt` | 27KB | Timing | LATER | Type-to-register mapping |
| `isg/me_chess.tof` | 749KB | Semantics | LOW | Type conversion table |
| `isg/me_regid.txt` | 4.7KB | Decoding | DONE | Register name-to-ID mapping |

### Files that do NOT matter (despite looking promising)

| File | Why it's useless |
|------|-----------------|
| `isg/*.ena` (1.4MB) | Binary DDDMP decision diagrams, not readable |
| `isg/me_chess.sfg` etc. | Compiler-internal scheduling graphs |
| `me_chess.tof` | Type signatures, not instruction formats |
| `me_pdg_encoding.p` | Does not exist |
| Entire amd-unified-software/ (175GB) | Vivado FPGA tools, no ISB/ISA data |

### Tools

| Tool | Path | Use |
|------|------|-----|
| `noodle` | `aietools/tps/lnx64/target_aie_ml/bin/LNa64bin/noodle` | ISB generator (needs nML source we don't have) |
| `xchesscc` | `aietools/bin/xchesscc` | Chess compiler + assembler oracle |
| `aiesimulator` | `aietools/bin/aiesimulator` | Behavioral comparison |

### nML Source

The nML processor description that generates everything (ISB, compiler,
simulator, RTL) is a Synopsys/AMD trade secret. We will never have it. But
we don't need it -- between the ISB, Python models, scheduling ISGs, and
aie-rt, we have all the data required for 100% compatible emulation. The
challenge is processing and integrating it, not acquiring it.

## Execution Priority

1. **NOW**: ISB parser completion (Pillar 1 -- close the decoding gap)
2. **NEXT**: Python model consumption (Pillar 2 -- vector semantics)
3. **THEN**: Configuration constants from .ctt (Pillar 2 -- mode selection)
4. **LATER**: Pipeline timing from ISG files (Pillar 3 -- cycle-accuracy)
5. **ONGOING**: Subsystem fixes as tests expose them (Pillar 4)

The path is clear. No information is missing. The work is integration.
