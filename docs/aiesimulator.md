# aiesimulator Integration Guide

This document covers how AMD's aiesimulator works, how xdna-emu integrates
with it, and what's required to get it running.

## What is aiesimulator?

aiesimulator is AMD's proprietary SystemC-based simulator for AIE arrays.
It models the full hardware -- cores, DMAs, locks, stream switches, NoC
interconnect -- at cycle-approximate fidelity. It is part of the aietools
installation (extracted from the Vitis/Unified Installer).

We use it as a **third validation point** alongside our emulator and real
NPU hardware. It is not an oracle (it has its own bugs), but divergence
between aiesimulator and our emulator is a strong signal that one of them
is wrong.

## Architecture

aiesimulator is a SystemC application. A typical invocation looks like:

```
aiesimulator --pkg-dir=aie.mlir.prj/sim --simulation-cycle-timeout=100000
```

### Components loaded at runtime

```
aiesimulator (bash wrapper)
    |
    v
aie2simmsm (actual SystemC binary, in bin/unwrapped/lnx64.o/)
    |
    +-- libaie2_cluster_msm_v1_0_0.osci.so   (AIE2 cluster model)
    +-- libsystemc.so                          (OSCI SystemC framework)
    +-- libxtlm.so                             (Xilinx TLM-2.0 bus model)
    +-- liblnoc_tlm.so                         (NoC interconnect model)
    +-- ps.so                                  (compiled host shim, per-design)
    +-- libxaienginecdo.so                     (CDO driver, from mlir-aie)
```

### The ps.so host shim

This is the critical piece that most people trip over. `ps.so` is a shared
library compiled from the test's host code (`test.cpp`) plus mlir-aie's
`genwrapper_for_ps.cpp`. It implements a SystemC module (`PSIP_ps_i3`) that:

- Presents AXI-MM bus interfaces to the simulated AIE array
- Runs the test's `ps_main()` function (which does buffer writes, lock
  operations, result checking)
- Links against `libxaienginecdo.so` for CDO command processing

The ps.so is compiled by `aiecc.py --aiesim` during the Chess build. Without
`--aiesim`, the build skips generating the simulation package entirely.

### libxaienginecdo.so

Built from `mlir-aie/third_party/aie-rt/driver/src/` as a static library
(`xaienginecdo_static`), then linked into `libxaienginecdo.so` during the
mlir-aie install. It lives at:

```
mlir-aie/install/runtime_lib/x86_64/xaiengine/lib/libxaienginecdo.so
```

When ps.so loads at simulation time, the dynamic linker needs to find this
library. The aiesimulator bash wrapper clobbers `LD_LIBRARY_PATH`, so we
work around it by symlinking `libxaienginecdo.so` into `sim/ps/` alongside
`ps.so` (see `ensure_xaiengine_symlink()` in `aiesimulator.rs`).

## Directory Structure

A simulation-ready Chess build produces this layout:

```
test_name/
    aie.mlir.prj/
        aiesim.sh               # runner script (generated by aiecc.py)
        main_core_1_3.elf       # compiled kernel
        ...
        sim/
            .target             # device target identifier ("hw")
            config/
                scsim_config.json   # SystemC simulator configuration
            arch/
                aieshim_solution.aiesol   # shim DMA port mapping
            reports/
                graph.xpe       # power/tile usage report
            ps/
                ps.so           # compiled host shim
                libxaienginecdo.so -> (symlink to mlir-aie install)
            flows_physical.mlir
            flows_physical.json
```

The `--pkg-dir` flag points to the `sim/` subdirectory, not the `.prj` root.
The `aiesim.sh` script handles this correctly.

## Build Flow

```
Input:  aie.mlir + test.cpp + kernel.cc
                |
                v
        aiecc.py --aiesim --xchesscc
                |
                +-- [1] xchesscc compiles kernel.cc -> .elf
                +-- [2] aie-translate generates sim config files:
                |       --aie-mlir-to-xpe         -> sim/reports/graph.xpe
                |       --aie-mlir-to-shim-solution -> sim/arch/aieshim_solution.aiesol
                |       --aie-mlir-to-scsim-config -> sim/config/scsim_config.json
                +-- [3] aie-find-flows             -> flows_physical.mlir
                +-- [4] aie-translate --aie-flows-to-json -> flows_physical.json
                +-- [5] clang++ compiles test.cpp + genwrapper_for_ps.cpp -> sim/ps/ps.so
                +-- [6] generates aiesim.sh
                |
                v
Output: aie.mlir.prj/sim/ (complete simulation package)
```

Step [5] is where `libxaienginecdo.so` is needed: `ps.so` links against it.
The clang++ invocation uses `-L{xaiengine}/lib -lxaienginecdo`.

## Running Peano-compiled cores (the Peano x aiesim cell)

aiesim is reputed to be "Chess-only", but that wall is an **aiecc build-flow
gate, not a simulator limitation**: `aiecc` refuses `--no-xchesscc --aiesim`
(aiesim "requires xbridge"; xbridge links only Chess objects). The closed AIE2
ISS (`aie2simmsm`) loads per-core ELFs and decodes the AIE2 ISA from the
instruction stream -- it does **not** require Chess metadata sections
(`.tctmemtab`, `.eoltab`, `.stackinfo`, `.rtstab`, `.chesstypeannotationtab`),
and the host `ps.so` is compiler-independent (`ps_main` comes from `test.cpp`).

So a Peano-compiled core runs through aiesim by **swapping it into a Chess-built
package**:

1. Chess `--aiesim` build the test -> a sound `aie.mlir.prj/sim` package; run it
   and confirm `PASS!` (baseline -- proves the package is good).
2. Peano-build the same kernel's core (`aiecc --no-xchesscc --no-xbridge`).
3. Overwrite `aie.mlir.prj/main_core_<col>_<row>.elf` with the Peano build
   (keep the Chess `ps.so`, `scsim_config.json`, flows) and re-run `aiesim.sh`.

Only the core ELF changes; the single variable is the kernel compiler. Buffer
and lock addresses come from mlir-aie's `AIEAssignBufferAddresses` pass, which
runs before either backend, so both compilers place them identically.

**Proven (2026-05-31):** scalar (`03_simple`) and vector (`vec_add`, genuine
`vadd.32` / `vldb` / `vst`) Peano cores both reach `PASS!` through aiesim. This
makes aiesim a **non-Phoenix-gated functional oracle** -- it validates Peano
*and* Chess output on any licensed box, with no silicon, and fills the
long-missing Peano column of the `{Peano,Chess} x {HW,EMU,aiesim}` grid.

**Scope / caveats:** aiesim is a *functional* oracle (value-correct); its timing
is cycle-*approximate*, not silicon-exact. The standalone swap is demonstrated
on single-tile inline kernels; multi-core / DMA / `kernel.cc` designs need
either lit-driven baselines or the planned **xclbin-boundary integration**
(drive aiesim from real bridge-test binaries via the emulator's own CDO parser,
which would also make aiesim a third bridge-test runtime). Working proof +
`swap_validate.sh` under `build/experiments/2026-05-31-aiesim-peano/`
(gitignored).

## Prerequisites

### License

aiesimulator requires a Xilinx/AMD feature license. The license file lives
at `~/.Xilinx/Xilinx.lic` and contains:

```
INCREMENT AIEMLsim xilinxd 2027.01 permanent uncounted ...
```

The license server also requires **network access** for verification -- the
FlexLM daemon phones home even for node-locked licenses. Both the license
file and network access must be available at simulation time.

### Environment

Required environment variables (set by `activate-npu-env.sh`):

| Variable | Value | Purpose |
|----------|-------|---------|
| `XILINX_VITIS_AIETOOLS` | `$NPU_WORK_DIR/aietools` | aietools root |
| `PATH` | includes `aietools/bin` | aiesimulator binary |
| `LD_LIBRARY_PATH` | includes xaiengine lib | libxaienginecdo.so |

**Important**: aietools libraries must be **appended** to `LD_LIBRARY_PATH`,
never prepended. aietools ships its own libstdc++ which is older than the
system version; prepending it shadows the system library and breaks LLVM
tools like `llvm-tblgen`.

### Sandbox incompatibility

**aiesimulator cannot run inside Claude Code's sandbox.** The sandbox blocks:

1. Read access to `~/.Xilinx/Xilinx.lic` (license file)
2. Network access for FlexLM license verification

Both are required. The test runner detects sandbox conditions and refuses to
invoke aiesimulator with a clear error message, rather than producing
misleading "license not found" failures.

## xdna-emu Integration

### Two execution paths

**`run_simulation()`** -- for npu-xrt style tests with separate I/O buffers:
- Creates `aiesim_input/` and `aiesim_output/` directories
- Passes `--input-dir`, `--output-dir` to aiesimulator
- Reads output via `read_output_data()`

**`run_unit_simulation()`** -- for chess_compiler_tests_aie2 unit tests:
- Runs `aiesim.sh` which handles I/O internally via ps.so
- Checks stdout for "PASS!" marker
- Simpler: no input/output dir management

Both paths:
- Symlink `libxaienginecdo.so` into `sim/ps/` before invocation
- Prepend xaiengine lib to `LD_LIBRARY_PATH`
- Auto-convert VCD traces to Perfetto JSON after completion

### CLI usage

```bash
# Run unit tests with aiesimulator (requires --compiler=chess or --chess-only)
cargo run --bin npu-test -- --suite=unit-tests --compiler=chess --runtime=aiesim

# Full validation grid with aiesim
cargo run --bin npu-test -- --runtime=emu,hw,aiesim --compiler=all --suite=all

# aiesim auto-enables chess compiler
cargo run --bin npu-test -- --runtime=aiesim
# note: aiesim requires Chess builds, enabling --compiler=chess
```

### runner.toml configuration

```toml
[aiesim]
# Behavioral config for aiesimulator (when activated via --runtime=aiesim).
enabled = false     # Legacy: use --runtime=aiesim instead

[unit_tests]
aiesim = true               # Run unit tests through aiesimulator
aiesim_timeout = 1_000_000  # Max simulation cycles
```

## Known Issues

### ERROR:101 spam (benign)

aiesimulator produces thousands of `[ERROR:101]: Address computes to a bank
value that doesn't exist` messages for unused tiles. The simulator probes all
tiles in the modeled device (VC2802 Versal, much larger than NPU1's 5x6
array). These errors are cosmetic -- the actual test tiles execute correctly.

### ps.so build requires aiecc.py --aiesim

If a Chess build was done without `--aiesim`, there is no `sim/` directory
and no `ps.so`. The test runner detects this and skips aiesim for that test.

### VC2802 device model

aiesimulator uses Versal device models (`VC2802.json`) even for NPU1 targets.
This is because the AIE2 tile architecture is identical between Versal and
NPU1 -- only the array dimensions and shim/NoC topology differ. The
`scsim_config.json` generated by mlir-aie maps the test's tile placement
onto the Versal model. Results are functionally equivalent.

### Cycle accuracy

Despite marketing, aiesimulator is "cycle-approximate", not cycle-accurate.
Its timing can diverge from real silicon, especially for:
- NoC latency (simplified model)
- Memory bank conflict resolution
- DMA pipeline depth

Use it for **functional correctness** validation, not timing analysis. For
timing, use real NPU hardware traces.

### Compute-to-compute shared memory is NOT modeled on AIE2 (hard limitation)

aiesimulator **cannot simulate compute-to-compute neighbour shared-memory / lock
handoff on AIE2/AIE-ML**. A producer core's lock release is never made visible to
the adjacent consumer core's neighbour-lock view, so any kernel that hands data
core-to-core through shared memory + locks (e.g. depth-1 objectfifos between
adjacent compute tiles) deadlocks. Same-tile locks and memtile locks work fine;
only the cross-core neighbour bridge is missing (the AIE2 ISS models core tiles in
isolation).

This is **AMD's limitation, not ours**: AMD's own `mlir-aie`
`chess_compiler_tests_aie2/04_shared_memory` is marked `XFAIL: *` (the AIE1
version passes), and AMD's native `aiesimulator` reproduces the failure standalone
(no bridge involved). Real silicon runs these kernels fine.

**Consequence:** exclude compute-to-compute neighbour-objectfifo kernels from the
aiesim oracle. In the corpus this is `matrix_multiplication_using_cascade` `buffer`
(the `cascade`/`plain` variants use cascade ports instead and pass). Our own Rust
emulator models direct neighbour-memory + locks, so it is *more* faithful than
aiesim for this class.

Full analysis, three-legged proof, and native repro:
[`docs/superpowers/findings/2026-06-06-aiesim-aie2-cross-core-shared-memory-limitation.md`](superpowers/findings/2026-06-06-aiesim-aie2-cross-core-shared-memory-limitation.md).
