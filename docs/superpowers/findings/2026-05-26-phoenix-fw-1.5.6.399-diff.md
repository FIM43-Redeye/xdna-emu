# Phoenix firmware 1.5.6.399 vs 1.5.5.391 diff

## Why

The 2026-05-25 telemetry-probe finding showed Phoenix
`MSG_OP_GET_TELEMETRY` returns 32 bytes of real per-channel slot
counters plus 1656 bytes of `0xff` placeholder -- not the cycle
ground truth the calibration mission wanted. Before committing to
the `MSG_OP_AIE_RW_ACCESS` fallback, worth checking whether a
newer Phoenix firmware (which we hadn't yet looked at) changes any
of the relevant handlers.

## How we found the new firmware

The xdna-driver build infrastructure pulls firmware blobs at
build time via the URL list in `xdna-driver/tools/info.json`.
For Phoenix the URL is:

    https://gitlab.com/kernel-firmware/drm-firmware/-/raw/amd-ipu-staging/
        amdnpu/1502_00/npu.sbin.1.5.5.391

That points at the `amd-ipu-staging` branch of the upstream
linux-firmware staging area. The directory listing (via GitLab
REST API) shows four Phoenix firmwares:

    1.4_npu.sbin.1.5.2.380
    npu.sbin.1.5.4.389
    npu.sbin.1.5.5.391
    npu.sbin.1.5.6.399          <- newest

There is no `1.6.x` Phoenix firmware in this branch -- the
`1.6.x` family belongs to Strix (NPU4) silicon.

## How to extract a body for analysis

`tools/fw-extract.py` strips the 256-byte signed header to produce
the 248080-byte Xtensa body that loads at `0x08ad3000`. The body
is what Ghidra analyzes. Self-test verified against the existing
project's `npu-fw-body.bin`.

## Methodology

1. Downloaded `npu.sbin.1.5.6.399` from amd-ipu-staging.
2. Extracted body via `fw-extract.py`.
3. Ran the existing Ghidra headless pipeline
   (`SetImageBase` -> auto-analyze -> `SeedFunctions` -> `DumpNpuFw`
   -> `DecompileNpuFw`) on the new body, producing a parallel
   `analysis/` directory.
4. Diffed function set, function bytes, and string set between
   the two analyses. Spot-checked telemetry-relevant addresses
   (`FUN_08ad9784` dispatcher, `FUN_08adae28` worker) at the byte
   level directly from the .body files.

## Findings

### Overall scope

| Metric | 1.5.5.391 | 1.5.6.399 | Diff |
|--------|-----------|-----------|------|
| .sbin total bytes | 248592 | 248592 | 10311 differ (4.1%) |
| body bytes | 248080 | 248080 | 10007 differ (4.0%) |
| Ghidra-recovered functions | 624 | 624 | 0 (just minor address shifts) |
| Strings | 340 | 340 | 0 (no add/remove, identical set) |
| Functions with byte-level change | -- | -- | 54 / 535 common |

### Where the 10007 bytes live

Of the 54 changed functions:

- **49** have only 1-2 byte differences, in a pattern consistent
  with literal-pool offset shifts (e.g. `FUN_08ad7470` changes
  15 bytes, every change is `+1` to an existing byte). These are
  Xtensa `l32r` immediates -- recompile-with-different-data-layout
  artifacts, not semantic changes.

- **1** function (`FUN_08ad5630`) has both a size change
  (586 -> 583 bytes) and an actual rewrite (446 / 586 bytes
  differ). Decompiles to a panic/fatal-error handler beginning
  with `rsil(2)` (Xtensa Read & Set Interrupt Level), called by
  13 sites all matching the pattern `FUN_08ad5630(); return;`.
  The shrink looks like an optimization of the panic path. Not
  telemetry/health/profiling related.

- **4** other functions have 3-4 byte diffs (`FUN_08ad8b08`,
  `FUN_08ad8480`, `FUN_08adaf50`, `FUN_08ad8dc0`) -- all
  consistent with literal-pool shifts.

### Telemetry path is byte-identical

Direct byte comparison of the telemetry call chain:

- `FUN_08adae28` (the GET_TELEMETRY worker that does
  `memset(buf, 0xff, 0x698)` + the 4-iteration `loop` that copies
  32 bytes of channel counters from `DAT_08ad6408+0x177`): **all
  117 bytes identical** between versions.
- `FUN_08ad9784` (the dispatcher that validates `buf_size > 0x697`
  and sets up DMA): **1 byte of 241 changed**, at offset +0xa7
  inside a `call8` immediate (target shifted slightly due to
  literal-pool re-layout). No semantic change.

### Bottom line

1.5.6.399 is a bug-fix / minor-cleanup release. There are no new
opcodes, no new strings, no new functions, no telemetry handler
changes, no APP_HEALTH wiring, no FW_LOG/FW_TRACE wiring. **A
firmware swap would not unlock any of the capabilities the
2026-05-06 NPU1 capability survey marked NOT IMPLEMENTED.**

## Implications for the calibration mission

Stay on 1.5.5.391. The `MSG_OP_AIE_RW_ACCESS` (0x203) path -- the
one op the survey confirmed Phoenix firmware actually implements
-- remains the route to trace-independent cycle ground truth.

## Open question: do later patch versions exist?

The `amd-ipu-staging` branch is the apparent canonical staging
area for AMD IPU firmware drops to linux-firmware. 1.5.6.399 is
the newest Phoenix entry there. No 1.5.7 / 1.5.8 visible. If AMD
has Phoenix firmware drops beyond 1.5.6.399, they are in a
channel we have not located (private builds, internal repos, or
a beta channel separate from this gitlab staging). Worth a more
targeted search if a later version becomes interesting -- but
even then, the version-bump-without-feature-change pattern here
suggests Phoenix is in maintenance mode and isn't going to grow
new capabilities.
