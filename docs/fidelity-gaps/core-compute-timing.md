# Core Compute Timing Gaps

Gaps in the compute core's per-cycle timing model -- specifically memory-port
stalls the hardware charges that the emulator does not. The core's *instruction
issue* timing (bundle-issue rate, lock function-call overhead) is faithful and
HW-validated; these gaps are in the memory-port / stall accounting layered on top.

| Gap | Model vs hardware | Where | Status / rationale |
|-----|-------------------|-------|--------------------|
| Compute-core MEMORY_STALL: per-access stall not modeled (count + cycle cost) | A memory-heavy compute loop (`eo[i]=ei[i]+1`, 32 elts x 16 reps) on real NPU1 fires **MEMORY_STALL ~220-244x per consumer core**; EMU fires **0**. EMU only models the intra-bundle structural bank-conflict pulse (`event_trace.rs:440`), not the broad per-access stall HW emits. **It is a cycle-cost gap, not cosmetic:** consA HW active span exceeds EMU's by +235cy ~= its un-modeled stall count (220 MEMORY_STALL + 17 extra LOCK_STALL), i.e. ~1 HW cycle per missing stall; the deficit back-pressures upstream (producer span +1235cy). Diagnosed via of_q0_rich EMU-vs-HW trace (PERF_CNT_2 interval 1024 both -> no clock-rate divergence; span deficit == stall count). | EMU core memory-port timing model; intra-bundle pulse `src/interpreter/state/event_trace.rs:440`; evidence `build/experiments/sp3-spike-trace/SP4A-EMU-FINDING.md` | **ROOT-CAUSED + MODELLED (bank-arbitration arc, 2026-07-14); superseded by the two rows below.** The "why" is answered: MEMORY_STALL is the core losing per-physical-bank round-robin arbitration against the tile DMA (AM020 ch.2:166), HW-confirmed cycle-by-cycle (every stall preceded by a bank conflict at t-1, 220/220 and 245/245). The emulator now models it and charges the cycle (`CoreStatus::WaitBank`); EMU no longer fires 0 but 425/440 vs HW's 220/245, with HW's exact stall shape. What remains is the magnitude, tracked as the DMA-cadence and backpressure-semantics rows below. Lock cycle-cost IS modeled (`737f5505`); only the lock *pulse* count is cosmetic. |
| DMA bank-demand density: EMU DMA requests a bank every cycle, HW bursts 16B per 4 beats | Task 8 validation of the bank-arbitration model against the Phoenix capture (`build/experiments/memory-stall-bankcap/`): the model gets the MECHANISM right (stalls on the consumers not the producer, physical banks 0/1 only, 96% singleton stalls at gap 2 -- HW's exact deny-and-retry shape) but **over-produces stall CYCLES**: ConsA 425 vs HW 220, ConsB 440 vs 245, Producer 102 vs 1. Also MM2S_0_MEMORY_STARVATION 40/53/128 vs HW 0. Root cause: `words_per_cycle_for` gives a streaming DMA channel 1 word (4B) per cycle, so `peek_bank_demand` declares a bank demand EVERY cycle of a transfer; silicon aggregates stream beats into 128-bit bank accesses (AM020 ch.2:164) and touches the bank once per ~4 beats, i.e. EMU's DMA demand is ~4x too dense and collides ~4x too often. Deliberately NOT fitted (design spec: idealized fast memory, DMA cadence varies with installed silicon). | `src/device/dma/engine/stepping.rs:1357` (`words_per_cycle_for`), `peek_bank_demand`; evidence `docs/superpowers/findings/2026-07-13-memory-stall-bank-arbitration.md` (Task 8 section), probe `of_q0_rich_bank_arbitration_vs_hw` | **OPEN (bank-arbitration arc, 2026-07-14).** Magnitude-only: mechanism, tiles, banks and shape all match HW. **Root cause now MEASURED on silicon, not hypothesised** (`docs/superpowers/findings/2026-07-14-dma-bank-access-width.md`, capture `build/experiments/bankwidth/`): a dedicated HW experiment pinned the tile DMA's bank-access width at **16.0 bytes = 4.00 stream beats** (Model B / per-beat predicts 4 B and is off by 3.7x). The DMA lost 112 bank arbitrations in the collide variant and paid 5 cycles for them -- it has memory-side slack and does not claim a bank every cycle. Fix = model the DMA's bank access as a 16-byte burst once per 4 stream beats, which moves DMA timing everywhere else in the emulator -- a separate scoped decision, not a stall-model edit. **Also measured:** `MM2S_0_MEMORY_STARVATION` is 0 on HW even under 112 lost arbitrations, so the emulator's emission of it on arbitration loss is semantically wrong, independent of the cadence fix. **Semantics now MEASURED too** (`docs/superpowers/findings/2026-07-14-dma-memory-pressure-event-semantics.md`, capture `build/experiments/s2mm-backpressure-semantics/`): both events are FIFO-occupancy signals. `S2MM_0_MEMORY_BACKPRESSURE` is 99.1% contained in `S2MM_0_STALLED_LOCK` windows, asserting exactly 15 cycles after each lock stall begins (zero variance over 39 windows) and deasserting 1 cycle after it ends -- the ingress FIFO fills in 15 beats while the BD cannot write. `BACKPRESSURE INTERSECT STREAM_STARVATION = 0` exactly, confirming both read the same FIFO. Bank contention (205 cyc) cannot explain 757 cyc of backpressure. Fix = model the DMA FIFO occupancy; both events fall out. |
| S2MM_MEMORY_BACKPRESSURE semantics: EMU emits it only on bank-arbitration loss | HW asserts `S2MM_0_MEMORY_BACKPRESSURE` for **756 / 853** cycles on the two consumers -- far more than the 232 / 255 cycles in which ANY bank was contended -- so on silicon the event cannot mean only "this channel lost a bank arbitration this cycle". The emulator emits it exactly and only for that (56 / 76 cycles): a strict subset of HW's assertion. The event evidently covers any cycle the S2MM write path cannot drain into memory, of which arbitration loss is one cause. | `src/interpreter/engine/coordinator.rs` Phase E (DMA bank-arbitration pressure block) | **OPEN (bank-arbitration arc, 2026-07-14).** Independent of the cadence gap above. Needs a HW experiment isolating what else backpressures the S2MM write port before the emulator can widen the emission -- widening it without that would be curve-fitting. |
| Bank-arbiter resume-cycle hole: one un-arbitrated bank touch on lock/DMA/stream resume | The per-physical-bank round-robin arbiter (bank-arbitration arc, `664f824e`) runs in the coordinator's Phase B demand peek (`CoreInterpreter::peek_bank_demand`), which declares nothing for a core in `WaitingLock`/`WaitingDma`/`WaitingStream` (`src/interpreter/core/interpreter.rs` ~198-205). But `try_resume_stall` (same file, ~858) resolves the stall INSIDE `step_internal`, on the commit path, one call after that cycle's peek already ran -- so the exact cycle a stalled core resumes, it decodes and executes its bundle having declared no bank demand for that cycle. The access reaches memory without ever reaching the arbiter: a **missed conflict, never a double grant** -- strictly an under-count of contention, never an over-count, and nothing is corrupted (EMU memory is not physically single-port). Measured unreachable on the validation corpus: disassembling 60 core ELFs (chess + peano, including the objectfifo kernels used for bridge validation) found **zero** bundles pairing a blocking op (`acq`, stream/cascade ops) with a load/store slot -- Peano always emits the acquire as its own out-of-line function (`acq r0, r1` / `ret lr`), a lone bundle with no memory slot, so nothing is ever both resuming AND touching a bank in the same bundle on the current corpus. | `src/interpreter/core/interpreter.rs`: `peek_bank_demand`'s `WaitingLock`/`WaitingDma`/`WaitingStream` exclusion (~198-205), `try_resume_stall` (~858) | **OPEN, dormant (task 6 review, bank-arbitration arc, 2026-07).** Zero bank touches leak on the current corpus (measured, above), so this is not an active fidelity regression today -- but a VLIW bundle CAN legally carry both a blocking op and a memory slot, so the hole is real, just unexercised. Fix is a non-mutating `can_resume_stall()` predicate consulted in the coordinator's Phase B demand peek instead of inside `step_internal`'s commit path; it needs the neighbour-lock snapshot for the `WaitingLock` arm (the only new plumbing) to evaluate the resume condition without mutating state. NOT implemented -- flagged for whoever next touches the stall-resume machinery, or if a kernel corpus ever produces a bundle that trips it. |

## Related: lock-arbitration cost (modeled) vs pulse emission (pending)

The per-lock-transaction arbitration **cycle cost** (HW trails every lock request
with a 1-cycle arbitration stall, even uncontended) is **modeled** in core cycle
timing (`737f5505`, `cycle_accurate.rs`; HW-exact on the tenant-4 core loop
2125->2129, full bridge corpus clean). What remains open is the *trace-event* half
-- emitting the per-transaction `LOCK_STALL` pulse on the uncontended path -- which
is a trace-encoding concern, cross-referenced from the aiesim "Trace
stall/starvation micro-timing" row in [`aiesim-oracle.md`](aiesim-oracle.md) and
the count-under-emission row in [`trace-encoding.md`](trace-encoding.md).
