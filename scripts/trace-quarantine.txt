# 2026-03-11: Permanent TDR -- control packet reconfiguration deadlock.
# ctrl_packet_reconfig and ctrl_packet_reconfig_4x1_cores TDR on real
# hardware (both compilers, serial execution, fresh reboot -- confirmed
# not a cascade).  Root cause: -aie-generate-column-control-overlay adds
# priority_route packet flows from shim DMA (MM2S) to tile TileControl
# ports.  When the application already has 4 packet_flow ops using the
# stream switch fabric, the overlay's control packets deadlock at runtime
# -- data flows wait for reconfiguration, control packets can't traverse
# congested fabric.  Evidence: ctrl_packet_reconfig_elf (same routing,
# no ctrlpkt.bin) passes; ctrl_packet_reconfig_1x4_cores (1 packet_flow
# instead of 4) passes.  This is a compiler/hardware limitation, not an
# emulator or tracing issue.

# 2026-05-10: ctrl_packet_reconfig_elf -- trace BD lands on the input BO,
# corrupting bo_in mid-run. The test uses ELF-mode kernel dispatch
# (xrt::ext::kernel with an xrt::elf module). XRT's relocation table only
# patches arg_idx=0..2 inside the ELF, which map to bo_in, bo_out, and the
# ctrl_pkt BOs respectively -- bo_trace (passed at kernel arg slot 5) has
# no relocation slot, so the trace BD15's source/dest address stays at
# whatever the MLIR baked in. That happens to be 0x800000008000, which
# overlaps the input BO at 0x9000 because the trace buffer is 1MB. Result:
# trace data overwrites the input partway through the run, the compute
# kernel reads garbage, and the test produces ~2500 / 4096 wrong outputs.
# HW passes because firmware-side BD address handling differs.
# Functional pass is restored by skipping trace injection on this test.
# Proper fix would require trace-prepare to inject bo_trace as a
# runtime_sequence argument (not just a host-side kernel arg) so XRT can
# relocate it. Deferred -- non-trivial cross-tool change.
ctrl_packet_reconfig_elf
